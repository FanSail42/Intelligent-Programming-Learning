### 3.2 上传「慢」问题排查与优化

用户感知慢主要在 **① 点击上传到提示成功**，而非后台 READY。

| 层级 | 问题 | 措施 |
|------|------|------|
| 前端 | 上传成功后 `await loadMaterials()` 阻塞 loading | 乐观插入行、后台刷新列表、立即结束 loading |
| 后端 | 响应前同步 dispatch | 改 `BackgroundTasks`，HTTP 先返回再入队 |
| 后端 | Celery `.delay()` 成功但无 Worker | 默认 `MATERIAL_DISPATCH_MODE=redis` + 进程内消费者 |
| 后端 | 跨课程同名资料重复向量化 | 同源文件 **克隆分块+向量**，跳过 Embedding |
| 后端 | Embedding 串行、分块过细 | 5 路并发批请求；chunk 768 / overlap 64 |
| 后端 | 仓库 ID 重复查库 | `warehouse_service` 内存缓存 |

**关键文件**：

- `backend/app/services/material_dispatch.py` — 统一分发（redis / thread / celery）
- `backend/app/services/material_queue.py` — Redis 队列消费者
- `backend/app/services/material_worker.py` — 线程池（最多 4 并发）
- `backend/app/services/material_pipeline.py` — 克隆快路径 + 耗时日志

## 四、Redis 与运行环境

### 4.1 问题链

1. Redis 未启动 / 防火墙拦截 → `get_redis()` 回退 `InMemoryRedis`
2. `InMemoryRedis` 缺 `brpop` → 队列消费者死循环报错
3. Redis 3.2 + `redis-py` 5.x 默认 RESP3 `HELLO` → 连接报 `unknown command 'HELLO'`
4. `redis.windows.conf` 含 `requirepass lifan`，`.env` 原无密码

### 4.2 处理措施

| 项 | 内容 |
|----|------|
| `InMemoryRedis` | 补全 `lpush` / `brpop`；无队列能力时回退线程池 |
| `get_redis()` | `protocol=2` 兼容 Redis 3.2 |
| `backend/.env` | `REDIS_URL=redis://:lifan@localhost:6379/0` |
| 防火墙 | `scripts/setup_redis_firewall.ps1` / `.bat`（允许 `redis-server.exe` 与 6379） |
| 启动 Redis | `scripts/start_redis.bat` → `D:\develop\Redis-x64-3.2.100` |

### 4.3 验证

- `redis-cli -a lifan ping` → **PONG**
- Python `get_redis().ping()` → **True**（真实 Redis 客户端）
