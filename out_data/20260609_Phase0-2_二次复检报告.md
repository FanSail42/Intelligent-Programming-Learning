## 五、迁移与 Schema 一致性（重点）

### 5.1 双轨迁移现状

```
Alembic: phase1 → phase2 → phase3 → phase4
运行时:  main.py startup → ensure_course_schema() + ensure_warehouse_schema()
```

| 问题 | 严重度 | 说明 |
|------|--------|------|
| phase4 Alembic **缺少** `warehouse_kind`、`course_subject` | **高** | 仅 `db_migrate` 启动时补列；纯 alembic 环境与模型不一致 |
| phase3 与 `db_migrate` 重复补审核列 | 中 | 已有库 `alembic upgrade` 可能报 Duplicate column |
| `ensure_course_schema` 假定 `course` 表已存在 | 高 | 空库直接 uvicorn 可能失败 |
| README 未强制 `alembic upgrade head` 顺序 | 中 | 新开发者易踩坑 |

### 5.2 推荐单一启动顺序

```text
1. docker compose up -d
2. cd backend && alembic upgrade head   # 若 phase3 已手动建列，可 stamp 后只跑 phase4
3. uvicorn app.main:app --reload        # db_migrate 补齐 warehouse_kind 等
4. python scripts/seed_demo.py
5. python scripts/seed_warehouses.py
6. （可选）python scripts/backfill_material_warehouse.py  # 建议新增：补 4 条空 warehouse_id
```
