## 0. 项目一句话定位

**慧编学伴**是面向高校计算机类课程（C/C++、Java、Python 等）的 Web 智能编程学习助教系统：以 **「课程 RAG 辅导 + 结构化代码讲解 + 规则化学情分析 + 教师班级视图 + 管理员 AI 治理」** 为主线，解决传统 OJ「只判对错」、通用 ChatGPT「脱离课程大纲」、教师「难以及时逐人答疑」三类痛点，而非替代在线判题系统。

**当前工程状态（截至 2026-06）**：Phase 0～4 核心功能已交付，后端 pytest **154 passed**，全项目综合完成度约 **84%**；生产部署模块 M08 待完善。

## 一、研究背景（撰写时需标注参考文献）

> 说明：下列段落为汇报正文，正式开题报告中请在句末标注对应文献序号。建议引用见每段末「可引文献」提示，完整条目见 `参考文献/参考文献_GB7714.txt`。

### 1.1 时代与政策背景

高等教育信息化与人工智能深度融合已成为全球高等教育改革的重要方向。我国持续推进教育数字化战略，鼓励将生成式人工智能、学习分析等技术应用于教学实践。计算机类专业作为培养数智时代核心人才的主阵地，数据结构、程序设计、Python 数据分析等课程普遍采用「课堂讲授 + 编程实验 + 在线评测（OJ）」模式。大语言模型（LLM）在代码理解、错误诊断与自然语言交互方面能力显著增强，为构建 **7×24 小时可及、可对话、可解释** 的智能编程助教提供了新的技术契机。

**可引文献方向**：生成式 AI 与高等教育数字化 [22]；LLM 编程教学趋势 [2][3][25]；教育数字化转型与个性化助教 [7]。

### 1.2 教学实践中的三类突出矛盾

结合本项目需求调研与现有文献，当前编程教学仍存在以下矛盾：

| 矛盾 | 具体表现 | 与本课题的关联 |
|------|----------|----------------|
| **师资与规模** | 班级扩大后，实验课中语法/逻辑/调试类问题难以及时逐人解答 | 需 AI 助教承担高频、重复性答疑 |
| **通用 AI 与课程脱节** | 学生直接使用 ChatGPT 等工具，答案常与教学大纲、课堂讲义、实验要求不一致，且难以核对来源 | 需 **课程级 RAG** 约束回答并展示引用 |
| **OJ 与形成性评价** | PTA/OJ 侧重「判对错」，对错误原因、调试思路、个性化复习路径支持不足 | 需 **结构化代码讲解 + 错题本 + 学情仪表盘** |

国内已有研究指出，LLM 可直接辅助代码解释与示例生成，但也存在答案不可靠、与课程要求不一致等问题，教育实践正从「直接调用通用聊天机器人」转向「领域知识约束 + 引用溯源 + 可控提示工程」的专用助教架构 [2][3][15]。同时，检索增强生成（RAG）可将生成回答锚定在本地知识库，缓解「幻觉」问题 [1][18][27]；学习分析则为刻画薄弱知识点与复习路径提供量化手段 [26]。

**可引文献方向**：LLM 编程教学困境与应对 [2][3][8][15][17]；生成式 AI 利弊 [19][20]；RAG 教育应用 [1][18][27]；智能助教平台 [7]。

### 1.3 现有方案与本课题的切入点

- **交互式学习平台**（Codecademy、Khan Academy 等）：强调即时反馈，但难以对接本校课程大纲与教师自建资料。
- **国内在线实验/OJ 平台**：自动判题成熟，自然语言辅导与个性化代码讲解能力有限。
- **通用 LLM 聊天工具**：零部署成本低，但缺乏课程知识隔离、引用溯源、班级学情与 AI 用量治理。

**研究切入点**：设计并实现一体化 Web 平台 **「慧编学伴」**，在吸收 LLM、RAG、学习分析等研究成果的基础上，强调 **多课程、多角色、可运维** 的端到端闭环——这正是现有单点系统或通用工具尚未充分覆盖的空白。

**可引文献方向**：AIOJ/智能编程训练 [13]；Python RAG 智能问答 [1]；实验指导 RAG 平台 [27]；代码评测智能体 [4]。

## 二、研究意义（撰写时不可标注参考文献）

### 2.1 理论意义

1. **探索 LLM + RAG 在高校编程教学中的系统化集成方式**  
   研究课程级向量隔离、检索相关性过滤、引用卡片展示与 SSE 流式输出如何协同，提升 AI 辅导的可信度与可追溯性，为「专用教育助教」而非「通用聊天机器人」提供可复现的工程范式。

2. **构建可解释、可测试的「AI 辅导—行为采集—学情反馈」闭环**  
   将代码讲解、对话行为、错题本、掌握度计算与规则推荐串联，采用公式化、规则化方法而非完全依赖 LLM 二次推理，为编程类学习分析提供可验证的实现路径。

3. **丰富多角色教育信息系统的设计经验**  
   在同一架构下贯通学生、教师、管理员三类角色及 AI 资源治理，为后续智慧教学平台扩展（作业管理、教师审核等）奠定架构基础。

### 2.2 实践意义

1. **对学生**：提供贴合所选课程的 AI 对话辅导、分级代码讲解、错题本与学习仪表盘，支持 7×24 小时自主复习与调试。
2. **对教师**：支持课程资料上传与知识库管理、班级学情概览（选课、错题 TOP、薄弱知识点、事件趋势），减轻重复答疑压力。
3. **对管理员**：提供账户管理、AI 模型配置、API 密钥加密存储与 Token 用量监控，保障系统可持续运维。
4. **对毕设答辩**：系统已具备可演示、可测试的完整样例（154 项自动化测试 + 演示种子脚本），具有明确的工程落地与展示价值。

## 三、研究内容

本课题以已设计并大部分实现的 **「慧编学伴」系统** 为研究对象，研究内容按功能模块与工程阶段组织如下。

### 3.1 需求分析与总体架构设计

- **角色与用例**：梳理学生、教师、管理员三类角色的核心用例（选课、辅导、讲解、学情、治理等）。
- **架构设计**：采用前后端分离 B/S 架构；后端 FastAPI + SQLAlchemy + JWT/RBAC；前端 Vue3 + Element Plus；数据层 MySQL（业务）+ Redis（会话/缓存/限流）+ Chroma（向量检索）。
- **规范设计**：统一 RESTful API 响应格式、错误码、模块文档（`docs/modules/Mxx_*.md`）与数据库逻辑外键/软删除规范。

### 3.2 课程知识库与 RAG 对话辅导（M03 + M04）

**研究重点**：教学资料如何工程化地变为「按课程隔离、可检索、可引用」的知识库。

| 环节 | 内容 |
|------|------|
| 资料管理 | 支持 PDF/TXT/MD/PPTX 上传；格式仓 + 课程仓；分派/克隆 |
| 解析流水线 | UPLOADED → PARSING → CHUNKING → EMBEDDING → READY；失败可重试 |
| 向量存储 | Chroma 按 **course_id** 隔离；chunk 元数据含页码/来源 |
| RAG 对话 | Top-K 语义检索 → **相关性过滤**（向量分 + 关键词重叠）→ Prompt 注入 |
| 交互体验 | SSE 流式多轮对话；回答展示 Top3 引用卡片；无命中时弱提示并可选记入错题本 |

**已实现**：资料流水线、RAG 检索与过滤、SSE 聊天、推荐问题等（模块完成度约 92%～95%）。

### 3.3 智能代码讲解（M05）

**研究重点**：如何将 LLM 能力约束为可分级展示的结构化教学反馈。

- 支持 C/C++、Java、Python 等语言代码提交（Monaco 编辑器）。
- 通过 Prompt 约束 LLM 输出 JSON：**syntax（语法）/ semantic（语义）/ runtime（运行）** 三级分析 + 参考代码。
- 严重/失败问题自动联动错题本（M06）。
- **说明边界**：运行级分析为静态推断，非 Judge0 沙箱真实执行——论文中需如实表述。

**已实现**：模块完成度 **100%**，含提交历史与前端分栏交互。

### 3.4 学习分析与推荐（M06）

**研究重点**：在不依赖 LLM 排序的前提下，实现可解释学情建模。

| 能力 | 实现思路 |
|------|----------|
| 事件埋点 | code_submit、code_analysis_error、chat_message、chat_no_context 等 |
| 错题本 | 自动归集、分类、ECharts 可视化、标记掌握 |
| 掌握度 | 时间衰减 + 事件权重公式（`mastery.py`） |
| 推荐 | 薄弱知识点 Top3 + 匹配 READY 状态课程资料（规则引擎） |
| 仪表盘 | 7 日趋势、薄弱 KP 阶梯图、近期活动；Redis 缓存 |

**已实现**：模块完成度约 **90%**。

### 3.5 教师教学支持与管理运维（M07 + M09 + M01/M02）

- **M02 课程与选课**：课程 CRUD、发布审核、学生选课/退课。
- **M07 教师班级学情**：选课人数/名单、班级错题 TOP、薄弱知识点、近 7 日事件趋势。
- **M09 账户与 AI 管理**：学生/教师账户 CRUD、AI 模型切换、Fernet 加密 API Key、日限额与 Token 用量趋势、操作日志。
- **M01 认证与个人中心**：JWT + Redis 黑名单；三类用户通用个人中心。

**已实现**：M07 约 85%，M09 约 98%；**M08 生产部署**（Docker 全栈/Nginx）约 15%，列入后续计划。

### 3.6 系统测试与综合评估

- pytest 自动化测试 **154 passed**（LLM/Embedding 统一 Mock，保证 CI 可重复）。
- 前后端联调、`seed_demo.py` / `seed_dashboard_demo.py` 答辩演示脚本。
- 功能测试、接口测试与模块验收文档对照。

## 四、拟解决的主要问题

以下五个问题对应本课题的核心技术难点，亦与系统已实现/待完善能力一一对应。

### 问题 1：课程语境下的可信 AI 辅导

**问题描述**：通用大模型易给出与课程讲义、实验要求不一致的答案，且学生无法核对依据。

**解决思路**：
- 教师上传资料 → 解析分块 → 按 **course_id** 向量化存储；
- 对话时仅检索当前课程知识库，并进行 **相关性过滤**；
- 前端展示 **引用来源卡片**（Top3），SSE 流式输出；
- 无有效检索命中时返回 `no_context` 提示，避免强行编造。

**对应实现**：`rag.py`、`rag_relevance.py`、`chat.py`、M03/M04。

---

### 问题 2：教学资料到可检索知识库的工程化流水线

**问题描述**：PDF/PPT 等格式多样，解析失败、向量化延迟、删除后向量残留等影响 RAG 质量。

**解决思路**：
- 状态机管理资料生命周期（含失败重试）；
- 统一分块与 Embedding 接口，Chroma collection 按维度分库；
- 软删除时同步清理向量；资料仓库支持跨课程复用与克隆。

**对应实现**：`material_pipeline.py`、`document_parser.py`、`chunking.py`、`vector_store.py`。

---

### 问题 3：超越 OJ 的代码教学反馈

**问题描述**：OJ 只告知「错/对」，缺少对语法、语义、运行逻辑的分层解释与可操作建议。

**解决思路**：
- Prompt 约束 LLM 输出 **结构化 JSON**（syntax/semantic/runtime）；
- Monaco 编辑器 + 参考代码一键复制；
- 分析结果写入错题本，与学习仪表盘联动。

**对应实现**：`code_analysis.py`、`prompts/code_analysis.yaml`、M05 + M06 钩子。

---

### 问题 4：可解释、可测试的学情建模

**问题描述**：若完全依赖 LLM 生成学习建议，成本高、不可复现、难以答辩验证。

**解决思路**：
- 统一学习事件模型，从代码讲解与 AI 对话行为自动采集；
- **规则公式** 计算知识点掌握度，**Top3 规则推荐** 匹配复习资料；
- ECharts 可视化 + Redis 缓存，保证仪表盘响应性能。

**对应实现**：`learning_events.py`、`mastery.py`、`recommendation.py`、`wrong_book_analysis.py`。

---

### 问题 5：多角色权限与 AI 资源治理

**问题描述**：多课程、多用户场景下，需隔离角色能力并防止 AI API 滥用与密钥泄露。

**解决思路**：
- JWT + RBAC 路由守卫与依赖注入；
- Redis 限流 `rate:llm:{userId}`、Token 日统计；
- 管理员配置 LLM/Embedding 模型，API Key **Fernet 加密** 存储，运行时热更新。

**对应实现**：M01 认证、M09 `runtime_ai_config.py`、`ai_usage.py`、`secret_crypto.py`。

## 五、主要创新点（供开题报告「创新点」章节直接使用）

1. **面向课程的知识隔离式 RAG 辅导机制**  
   向量检索 + 相关性过滤 + 引用卡片 + SSE 流式输出一体化，强化答案与课程资料的可追溯性。

2. **「代码讲解—错题本—掌握度—推荐」规则闭环**  
   以行为数据驱动、公式可解释，降低对 LLM 二次调用的依赖，适合毕设阶段的可靠实现与测试。

3. **三级结构化智能代码讲解**  
   语法/语义/运行分层 JSON 反馈 + Monaco 交互，弥补 OJ 在过程化辅导上的不足。

4. **兼顾教学与管理的多角色一体化平台**  
   学生端 + 教师班级学情 + 管理员 AI/账户治理同仓交付，形成可演示、可测试的完整工程样例。
