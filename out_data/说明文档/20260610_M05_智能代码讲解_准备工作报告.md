## 二、模块定位与验收目标

### 2.1 业务定位

智能代码讲解是系统与 OJ「只判对错」的 **核心差异化能力**：学生提交源码后，系统返回 **可理解的分级讲解**（语法 / 语义 / 运行），含问题定位、修改建议与示例，而非单一 pass/fail。

### 2.2 P0 最低交付（来自 `02_skill.md` §4）

| 类别 | 交付物 |
|------|--------|
| 数据库 | `code_submission`、`analysis_result` |
| 后端 API | `POST /api/v1/code/submit`、`GET /api/v1/code/submit/{id}/result`（历史列表为路线扩展项） |
| 前端 | `/student/code` + 代码编辑器 + 分级结果展示 |
| 测试 | `test_code_analysis.py`（Mock LLM 结构化 JSON） |
| 演示 | 提交含错 Python 代码 → 展示分级讲解 |

### 2.3 与 M04 的边界

| 维度 | M04 AI 对话 | M05 代码讲解 |
|------|-------------|--------------|
| 输入 | 自然语言问题 | 结构化源码 + 语言枚举 |
| 上下文 | RAG 检索课程切片 | **以源码为主**；可选注入课程知识点（P1） |
| 输出 | SSE 流式 Markdown | **结构化 JSON**（Pydantic 校验后落库） |
| 权限 | `ensure_chat_access` | 学生须 **已选课**（同 M04 学生侧） |

## 四、技术方案建议（开工设计稿）

### 4.1 数据库设计（建议 DDL 草案）

#### `code_submission`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 自增 |
| `user_id` | BIGINT | 逻辑外键 → `user.id` |
| `course_id` | BIGINT | 逻辑外键 → `course.id` |
| `assignment_id` | BIGINT NULL | Phase 4 作业域预留 |
| `language` | VARCHAR(16) | 枚举：`python` / `java` / `cpp`（MVP 先 `python`） |
| `source_code` | TEXT | 原始源码 |
| `version` | INT DEFAULT 1 | 同用户同课多次提交递增（可选 P1） |
| `deleted` | TINYINT | 软删除 |
| `created_at` / `updated_at` | DATETIME | 与现表一致 |

**索引建议**：`(user_id, course_id, created_at)`、`(course_id, created_at)`。

#### `analysis_result`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 自增 |
| `submission_id` | BIGINT | 逻辑外键 → `code_submission.id`，**1:1** |
| `status` | VARCHAR(16) | `pending` / `running` / `done` / `failed` |
| `result_json` | JSON NULL | 结构化讲解（见 §4.2） |
| `error_message` | VARCHAR(512) NULL | LLM/解析失败原因 |
| `deleted` | TINYINT | 软删除 |
| `created_at` / `updated_at` | DATETIME | — |

**迁移规范**：仅通过 **新 Alembic 版本** 建表；同步更新 `docs/database.md`（`02_skill.md` §7 纠偏：禁止再扩展 `db_migrate.py` 补列模式）。

### 4.2 结构化结果 Schema（Pydantic 草案）

与 `05` 号报告、`06` 路线对齐，建议三级 Tab 对应同一 JSON：

```json
{
  "summary": "一句话总评",
  "levels": {
    "syntax": {
      "score": "ok|warning|error",
      "issues": [
        { "line": 3, "message": "缩进不一致", "hint": "检查 elif 块缩进" }
      ]
    },
    "semantic": {
      "score": "ok|warning|error",
      "issues": [
        { "message": "循环条件可能永不结束", "explanation": "..." }
      ],
      "suggestions": ["将 while True 改为带退出条件"]
    },
    "runtime": {
      "score": "ok|warning|error",
      "issues": [
        { "message": "可能 IndexError", "explanation": "..." }
      ],
      "stderr_hint": null
    }
  },
  "fixed_code": "可选的修正示例",
  "examples": ["常见错误对比示例"]
}
```

**MVP 策略**：

- **语法 / 语义**：LLM + Prompt 约束 JSON 输出  
- **运行级**：答辩文案写「语义级讲解」；若未接沙箱，**禁止声称已真实执行**  
- **加分项**：`httpx` 调 Judge0 或 `subprocess` 沙箱，将 `stderr` 写入 `runtime.stderr_hint`

### 4.3 Prompt 设计要点

新建 `backend/prompts/code_analysis.yaml`，建议包含：

1. 角色：编程助教，面向初学者  
2. 输出：**仅 JSON**，字段与 Pydantic 一致  
3. 语言：按 `language` 字段切换讲解术语（Python 优先）  
4. 安全：不执行学生代码；不建议危险 API  
5. 长度：源码超过 N token 时只分析前段并注明截断  

可参考 `chat_rag.yaml` 的分步讲解风格，但输出改为 JSON 而非 Markdown。

### 4.4 API 设计（草案）

| 方法 | 路径 | 角色 | 说明 |
|------|------|------|------|
| POST | `/api/v1/code/submit` | student（已选课） | Body: `course_id`, `language`, `source_code`；创建 submission + analysis(pending)，触发分析 |
| GET | `/api/v1/code/submit/{id}/result` | 提交者 / admin | 返回 submission + analysis 状态与 `result_json` |
| GET | `/api/v1/code/submissions` | student | 可选 P1：`?course_id=` 历史列表分页 |

**业务规则**：

1. `course_id` 必填；`ensure_student_enrolled`  
2. `source_code` 非空；长度上限建议 **16KB**（配置项 `code_max_source_chars`）  
3. `language` 白名单；MVP 仅 `python` 返回 40001  
4. 每次分析调用 `check_llm_rate_limit`  
5. LLM 返回非 JSON → `status=failed`，`error_message` 落库  

**同步 vs 异步**：

| 方案 | 适用 | 实现 |
|------|------|------|
| **同步（推荐 MVP）** | 答辩演示、pytest 简单 | POST 内 await 分析，60s 超时，直接返回 `done` |
| 异步（P1） | 大模型慢、Java 编译 | Celery `analyze_code_task` + 轮询 GET result |

### 4.5 前端方案

| 项 | 建议 |
|----|------|

| 导航 | `MainLayout.vue` 学生菜单增加「代码讲解」 |

**依赖**：当前 `package.json` **无 Monaco**，需在 Phase 3 开工日安装并锁定版本。

## 六、风险 register

| ID | 风险 | 等级 | 缓解措施 |
|----|------|------|----------|
| R1 | LLM 不遵守 JSON 格式 | 高 | Pydantic 校验 + 一次 repair prompt；失败落 `failed` |
| R2 | 「运行级」无真实执行被质疑 | 中 | MVP 文案降级为「静态推断」；Judge0 作加分 |
| R3 | 同步分析超时（>60s） | 中 | httpx timeout；前端 Loading；后续 Celery |
| R4 | Monaco 体积与 Vite 配置 | 低 | 按需加载语言包；或 MVP textarea |
| R5 | 多语言 Prompt 质量参差 | 中 | P0 仅 Python；Java/C++ 标为 P1 |
| R6 | 代码含敏感/恶意内容 | 中 | 复用 `validate_user_input`；不上传执行 |
| R7 | 费用与 Token 膨胀 | 中 | 限流 + 源码截断 + `estimate_tokens` |
